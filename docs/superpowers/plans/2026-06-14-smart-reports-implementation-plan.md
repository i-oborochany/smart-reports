# Умные отчёты — план реализации

- **Дата:** 2026-06-14
- **Спецификация:** [2026-06-14-smart-reports-design.md](../specs/2026-06-14-smart-reports-design.md)
- **Стек:** Node.js + TypeScript (NestJS), React + TypeScript, PostgreSQL, Redis (BullMQ), Docker Compose, nginx + Let's Encrypt.
- **Хостинг:** FirstVDS (РФ — близко к API МС, помогает с 152-ФЗ).

## Принцип планирования

Идём вертикальными срезами: как можно раньше получаем **рабочий сквозной отчёт в Telegram**, затем расширяем формы, UI и «умные» функции. Каждая фаза заканчивается проверяемым результатом (Definition of Done).

---

## Структура репозитория (monorepo)

```
smart-reports/

│   └── deploy/       # скрипты деплоя на FirstVDS
└── docs/
```

## Фаза 0 — Фундамент и инфраструктура

**Цель:** пустой, но рабочий и развёртываемый каркас.

- [ ] Инициализировать monorepo (pnpm workspaces), линтер, prettier, tsconfig.
- [ ] Каркас NestJS (`apps/api`): конфиг через env, health-check.
- [ ] Каркас React (`apps/web`, Vite): пустая страница, сборка.
- [ ] PostgreSQL + ORM (Prisma) с первой миграцией; Redis.
- [ ] `docker-compose.yml`: postgres, redis, api, web, nginx.
- [ ] nginx + TLS (Let's Encrypt), привязка домена.
- [ ] Скрипт деплоя на FirstVDS (pull → build → up), бэкап БД (pg_dump по cron).
- [ ] CI: lint + build + test на push.

**DoD:** по адресу `https://<домен>` отдаётся страница `web`, `/api/health` отвечает, всё поднимается одной командой на FirstVDS.

## Фаза 1 — Интеграция с МоимСкладом

**Цель:** приложение устанавливается в МС и получает доступ к данным аккаунта.

- [ ] **Vendor API:** эндпоинты установки/активации/удаления приложения; получение и хранение `access_token` аккаунта (шифрование at-rest).
- [ ] Проверка подписи/контекста iframe (JWT от МС) для аутентификации пользователя.
- [ ] Модель `Account` (многотенантность), статус подписки.
- [ ] **JSON API 1.2 клиент:** базовый HTTP-клиент с троттлингом под лимиты МС, ретраи с бэкоффом, пагинация.
- [ ] Справочники из МС: организации, каналы(?)/проекты, склады, точки, сотрудники, контрагенты, группы товаров — для конструктора фильтров (с кэшем).

**DoD:** тестовый аккаунт МС «устанавливает» приложение, токен сохранён, клиент успешно тянет, например, список организаций и продажи за вчера.

## Фаза 2 — Ядро: модель данных и конструктор отчётов

**Цель:** из конфигурации отчёта получить посчитанные цифры.

- [ ] Схема БД: `Report` (10 полей анатомии), `Recipient`, `Schedule`, `Snapshot`, `DeliveryLog`, `Signal`, `Plan`.
- [ ] Описать конфиг отчёта как типизированную схему (в `packages/shared`).
- [ ] **Движок конструктора:** метрики × группировка × фильтры (вкл/искл) × период → маппинг на эндпоинты МС (`report/sales`, `report/profit` для старта) → агрегация и разбивка.
- [ ] Поддержать фильтры: каналы, проекты, организации, ответственный-сотрудник, контрагенты, склады/точки, группы/типы товаров.
- [ ] Юнит-тесты движка на фикстурах ответов МС.

**DoD:** для конфига «отгрузки за период с разбивкой по каналам, исключая контрагента X» движок возвращает корректные суммы (проверено на тестовых данных).

## Фаза 3 — Telegram-бот и первый сквозной отчёт (🎯 MVP)

**Цель:** один отчёт приходит в Telegram по расписанию.

- [ ] Единый бот (grammY): `/start`, привязка получателя по коду.
- [ ] Модель `Recipient` + привязка `chatId`.
- [ ] Рендер **текст-карточки** с эмодзи из результата движка.
- [ ] Планировщик на BullMQ: расписание (день/неделя/месяц/год) + часовой пояс аккаунта → задача → формирование → отправка.
- [ ] Логирование доставки (`DeliveryLog`), ретраи при сбое.

**DoD:** включённый отчёт «Отгрузки за вчера» в заданное время сам приходит карточкой в Telegram привязанному получателю. **Это первый демонстрируемый результат.**

---

## Фаза 4 — Формы доставки и архив

**Цель:** PDF/Excel/PNG, e-mail, история.

- [ ] Рендер **Excel** (ExcelJS) — табличные отчёты.
- [ ] Рендер **PDF** (Puppeteer) — сводки.
- [ ] **PNG-графики** (для трендов).
- [ ] Доставка по **e-mail** (SMTP) с вложениями.
- [ ] Сохранение сформированных отчётов в архив (файлы + метаданные).
- [ ] Кнопка/команда **«Отправить сейчас»** (тест до расписания).

**DoD:** отчёт можно получить в любой комбинации форм (текст+PDF+Excel) в Telegram и на e-mail; он сохранён в архиве.

---

## Фаза 5 — Веб-интерфейс (iframe в МС)

**Цель:** настройка отчётов без кода, удобно руководителю.

- [ ] Аутентификация по контексту МС, layout приложения.
- [ ] Список отчётов, **сгруппированный по группам**, вкл/выкл переключателем.
- [ ] **Редактор отчёта** = UI конструктора (метрики, группировка, фильтры с поиском по справочникам, период, сравнение, форма, расписание).
- [ ] Управление **получателями** (создать, выдать код привязки, e-mail; тип общий/личный).
- [ ] **Лента и архив**: онлайн-просмотр сформированного отчёта, скачать, переслать.

**DoD:** пользователь полностью настраивает и запускает отчёт из интерфейса внутри МС, видит архив.

---

## Фаза 6 — Умные функции (A–E + заработок)

**Цель:** превратить цифры в смысл.

- [ ] **Снимки агрегатов:** ежедневный фоновый сбор ключевых метрик по аккаунту.
- [ ] **База сравнения:** прошлый аналогичный период / среднее за период / план → ±% и стрелки 🟢🔴.
- [ ] **План/факт:** модель `Plan`, % выполнения.
- [ ] **Топы и антирейтинги** в отчётах.
- [ ] **Заработок сотрудника:** правило «% от метрики в рамках …» → личная сумма; режим «личной» рассылки (каждому свой срез).
- [ ] **AI-комментарий:** короткий вывод по отчёту (через Claude API).

**DoD:** в карточке отчёта есть сравнение со выбранной базой, топы, а у личного отчёта сотрудника — его заработок; есть AI-резюме.

---

## Фаза 7 — Сигналы / алерты

**Цель:** push, когда что-то важно, вне расписания.

- [ ] Движок порогов и правил (крупная продажа, ниже себестоимости, товар закончился, падение выручки, крупный возврат, просрочка платежа).
- [ ] Источники событий: вебхуки МС + периодический опрос там, где вебхуков нет.
- [ ] Реал-тайм отправка алертов получателям; настройка порогов в UI.

**DoD:** при наступлении события (на тестовых данных) получатель сразу получает алерт.

---

## Фаза 8 — Базовые пресеты (11)

**Цель:** «поставил → включил → получаешь».

- [ ] Реализовать 11 предустановленных отчётов как пресеты конструктора с дефолтными расписаниями (раздел 7 спецификации).
- [ ] Сидинг пресетов при установке приложения (выключены, ждут включения).
- [ ] Проверить каждый на тестовом аккаунте.

**DoD:** на свежем аккаунте после установки видны 11 готовых отчётов, любой включается в 2 клика + получатель.

---

## Фаза 9 — Подготовка к маркетплейсу и эксплуатация

**Цель:** готовность к публикации и нагрузке.

- [ ] Биллинг/подписка через МС (тарифы — когда решим монетизацию).
- [ ] Онбординг и страница приложения в маркетплейсе.
- [ ] Мониторинг (логи, метрики, алерты падений), бэкапы БД, ротация.
- [ ] Нагрузочное тестирование, распределение расписаний во времени под многотенантность.
- [ ] Политика приватности / 152-ФЗ, документация.

**DoD:** приложение проходит ревью маркетплейса МС и стабильно работает на проде.

## Риски и заметки

- **Лимиты API МС** — главный технический риск тяжёлых отчётов; смягчаем троттлингом, очередями, снимками агрегатов и разнесением расписаний.
- **Vendor API нюансы** (установка, биллинг) — уточнить по актуальной документации МС на Фазе 1.
- **Dropbox + git** — для разработки лучше работать из обычной папки/клона, а не из синхронизируемой.
- **Монетизация** — отложена; влияет только на Фазу 9 и на разбивку бесплатно/платно.

## Открытые вопросы к старту разработки

1. Домен для приложения (купить/выбрать).
2. Параметры VPS на FirstVDS (создать сервер под dev/prod).
3. Доступ к тестовому аккаунту МС и регистрация черновика приложения в кабинете разработчика МС.
